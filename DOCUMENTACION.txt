DESCRIPCION GENERAL
================================================================================

DownLabs.Core.Api es una API RESTful desarrollada en .NET 9.0 que sirve como 
backend para la plataforma DownLabs. Utiliza Supabase (PostgreSQL) como base de 
datos y sigue el patron de Minimal APIs de ASP.NET Core.

El proyecto permite gestionar:
- Clientes (empresas que solicitan cotizaciones)
- Mayoristas (proveedores de productos)
- Operadores (personal de DownLabs que procesan solicitudes)
- Catalogo de Productos
- Solicitudes de Cotizacion
- Cotizaciones generadas
- Pedidos con Credito


ARQUITECTURA DEL PROYECTO
================================================================================

El proyecto sigue una arquitectura limpia organizada en capas:

/Models
    Contiene las clases que mapean a las tablas de Supabase:
    - Cliente.cs          -> tabla "clientes"
    - Mayorista.cs        -> tabla "mayoristas"  
    - Operador.cs         -> tabla "operadores"
    - CatalogoProducto.cs  -> tabla "catalogo_productos"
    - SolicitudCotizacion.cs -> tabla "solicitudes_cotizacion"
    - CotizacionDownlabs.cs -> tabla "cotizaciones_downlabs"
    - PedidoCredito.cs    -> tabla "pedidos_creditos"

/Services
    Logica de negocio reutilizable:
    - ICrudService.cs     -> Interfaz generica para operaciones CRUD
    - CrudService.cs      -> Implementacion que se conecta directamente 
                            a Supabase usando la biblioteca oficial

/Endpoints
    Define los endpoints de la API para cada recurso:
    - AuthEndpoints.cs        -> Registro y login de usuarios
    - ClienteEndpoints.cs
    - MayoristaEndpoints.cs
    - OperadorEndpoints.cs
    - CatalogoProductoEndpoints.cs
    - SolicitudCotizacionEndpoints.cs
    - CotizacionDownlabsEndpoints.cs
    - PedidoCreditoEndpoints.cs

Cada archivo de endpoints sigue el mismo patrón de diseño.


AUTENTICACIÓN
================================================================================

El sistema usa Supabase Auth para gestionar usuarios:

Registro (/api/auth/register):
- Email, password, rol (cliente/mayorista/operador)
- Para clientes/mayoristas: nombre_empresa
- Para operadores: nombre, apellido
- Crea automáticamente registro en tabla correspondiente

Login (/api/auth/login):
- Email y password
- Retorna: access_token, user_id, email, rol

El campo "rol" determina en qué tabla se crea el usuario:
- "cliente" -> tabla clientes
- "mayorista" -> tabla mayoristas
- "operador" -> tabla operadores


FLUJO DE DATOS
================================================================================

1. Cliente -> Solicita Cotizacion
   Un cliente envia una solicitud de cotizacion con:
   - ID del cliente
   - Descripcion del articulo
   - Nivel de urgencia (Normal/Urgente)

2. Operador -> Procesa Solicitud
   Un operador recibe la solicitud y puede:
   - Cambiar el estado (Pendiente/Cotizada/Rechazada)
   - Asignarse a la solicitud (id_operador)

3. Sistema -> Genera Cotizacion
   Se crea una cotizacion vinculada a:
   - La solicitud original (id_solicitud)
   - Un producto del catalogo (id_producto)
   - Precio final para el cliente
   - Costo de envio
   - Estado (Negociando/Aceptada/Rechazada)

4. Cliente -> Genera Pedido
   Si la cotizacion es aceptada, se puede crear un pedido de credito:
   - Vinculado a la cotizacion (id_cotizacion)
   - Indica si requiere credito
   - Monto total incluyendo financiamiento
   - Estado de pago (Pendiente/Pagado/En mora)
   - Fechas de inicio y vencimiento del credito
   - Tipo de pago (contado/credito_30/credito_60/transferencia)


PATRON DE DISENO DE ENDPOINTS
================================================================================

Cada recurso sigue el mismo patron REST:

GET    /api/recurso           -> Listar todos (con paginacion)
GET    /api/recurso/{id}     -> Obtener por ID
POST   /api/recurso          -> Crear nuevo
PUT    /api/recurso/{id}     -> Reemplazar completamente
PATCH  /api/recurso/{id}     -> Actualizar parcialmente
DELETE /api/recurso/{id}    -> Eliminar

PARAMETROS DE CONSULTA:
- page      -> Numero de pagina (default: 1)
- pageSize  -> Items por pagina (default: 20, max: 100)
- search    -> Buscar en campos de texto
- filtros especificos por recurso


SERVICIO CRUD GENERICO
================================================================================

El CrudService es un servicio singleton que maneja todas las operaciones
de base de datos de manera generica:

Metodos disponibles:
- GetAllAsync<T>(tableName)        -> Obtiene todos los registros
- GetByIdAsync<T>(tableName, idColumn, id) -> Obtiene uno por ID
- CreateAsync<T>(tableName, data)  -> Inserta nuevo registro
- UpdateAsync<T>(tableName, idColumn, id, data) -> Actualiza
- DeleteAsync(tableName, idColumn, id) -> Elimina

Este servicio:
- Se conecta directamente a Supabase via HTTP
- Serializa/deserializa JSON automaticamente
- Maneja errores de base de datos
- Soporta CancellationToken para operaciones async


CONEXION A SUPABASE
================================================================================

La conexion se configura en appsettings.json:

{
  "Supabase": {
    "Url": "https://gszxuzkibrdzmklhnykv.supabase.co",
    "Key": "clave-api-supabase"
  }
}

La biblioteca Supabase.NET se encarga de:
- Autenticacion con la API key
- Transformar consultas REST a SQL
- Manejar tipos de datos PostgreSQL
- Soporte para PostgreSQL features (JSON, arrays, etc.)

================================================================================

{
  "success": true,
  "data": [...],
  "pagination": {
    "page": 1,
    "pageSize": 20,
    "total": 100,
    "totalPages": 5,
    "hasNext": true,
    "hasPrev": false
  }
}


MANEJO DE ERRORES
================================================================================

Formato consistente de errores:

{
  "success": false,
  "error": "NotFound",
  "message": "Descripcion del error"
}

Tipos de error:
- ValidationError (400) -> Datos invalidos en la peticion
- NotFound (404)       -> Recurso no encontrado
- BadRequest (400)     -> Error de logica de negocio
- InternalError (500)   -> Error interno del servidor

================================================================================

2. Ejecutar: dotnet run

4. Endpoints de prueba en: pruebas/api.http (para VS Code REST Client)
